# n2n edge 获取与 supernode 延迟说明

## 1. 如何拿到 edge.exe

`edge.exe` 是 n2n 在玩家电脑上运行的客户端程序。联机助手第一版不自动下载第三方二进制文件，原因是 VPN/虚拟网卡类程序需要明确来源、版本和安全边界。

推荐优先级：

### 方案 A：从官方源码自行构建，推荐

从 `ntop/n2n` 官方源码构建 Windows 版 edge，得到 `edge.exe` 后放到：

```text
tools/n2n/edge.exe
```

优点：

- 来源最清晰；
- 可以固定版本；
- 后续能记录构建参数和哈希；
- 更适合将来打包进安装器。

缺点：

- 第一次配置构建环境较麻烦；
- 需要处理 Windows 编译链。

### 方案 B：使用可信构建产物，适合临时测试

如果官方 release 没有直接提供 Windows exe，可以先使用可信来源的 Windows 构建产物做联调测试。

要求：

- 只用于开发测试；
- 记录下载地址、版本、SHA256；
- 不要直接把来历不明的 exe 打进正式安装包；
- 后续正式分发前改为自建构建流水线。

### 方案 C：让联机助手后续增加“下载/安装 n2n”功能

这是后续功能，不是当前第一版默认能力。

需要实现：

- 固定版本；
- 下载地址白名单；
- SHA256 校验；
- 用户明确确认；
- 下载失败和杀毒拦截提示；
- 安装/卸载说明。

## 2. 放置方式

当前客户端会检测：

```text
tools/n2n/edge.exe
tools/n2n/n2n.exe
```

只要其中一个存在，网络配置页就会认为 n2n edge 可用。

## 3. VPS 很远会不会导致延迟高

要分两种情况。

### 情况 A：P2P 直连成功

如果两个 edge 最终成功直连，supernode 主要负责发现和协调。

这时游戏数据大致是：

```text
玩家 A  ←→  玩家 B
```

VPS 很远通常只影响：

- 初始发现速度；
- 保活；
- 重新连接；
- 少量控制流量。

对游戏延迟影响较小。

### 情况 B：P2P 失败，流量经过 supernode 中继

如果双方 NAT/防火墙导致无法直连，流量可能绕 supernode。

这时游戏数据大致是：

```text
玩家 A  →  远距离 VPS  →  玩家 B
```

延迟会明显升高，近似受：

```text
玩家 A 到 VPS 的延迟 + VPS 到玩家 B 的延迟
```

影响。

如果 VPS 在海外，而两个玩家都在国内，游戏体验可能明显变差。

## 4. VPS 选择建议

优先选择：

- 离大多数玩家都近的区域；
- 网络线路稳定的 VPS；
- UDP 不受限的 VPS；
- 带宽和流量足够的 VPS。

小范围朋友联机建议：

```text
两个人都在中国大陆：优先国内或近距离亚洲节点
一个国内一个海外：选择双方延迟折中的地区
都在同一海外区域：选择该区域附近 VPS
```

## 5. 联机助手后续需要增加的检测

后续应在客户端中增加：

- supernode UDP 可达性测试；
- edge 是否启动成功；
- 虚拟 IP 是否分配成功；
- 两个虚拟 IP 是否互 ping；
- 游戏端口 TCP/UDP 是否可达；
- 尽量判断当前连接是 P2P 直连还是中继；
- 对多个 supernode 做延迟评分。

当前阶段先实现：

- edge 放置检测；
- 配置保存；
- edge 启停；
- 虚拟 IP 读取；
- 游戏端口连通性测试。
