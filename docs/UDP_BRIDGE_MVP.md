# UDP 端口代理与 UDP 广播桥 MVP 设计

更新时间：2026-06-03

本文档用于区分并规划联机助手后续两类 UDP 能力：UDP 单播端口代理与 UDP 广播桥。两者解决的问题不同，不能混为一个功能。

## 1. 背景

当前项目已经实现 TCP 端口代理，可以解决“游戏服务只监听 127.0.0.1 或只暴露本机端口，朋友需要通过虚拟 IP 访问”的一部分问题。

但很多局域网游戏使用 UDP：

- 有些游戏使用 UDP 单播端口传输数据；
- 有些游戏使用 UDP 广播或组播做房间发现；
- 有些游戏 TCP/UDP 混用；
- 有些游戏发现阶段用广播，加入阶段仍可 IP 直连。

因此 UDP 能力必须分成两个阶段做。

## 2. UDP 单播端口代理

### 2.1 解决什么问题

UDP 单播端口代理用于这种场景：

```text
朋友 -> 房主虚拟 IP:对外 UDP 端口 -> 联机助手 UDP 代理 -> 127.0.0.1:游戏 UDP 端口
```

适用情况：

- 游戏使用固定 UDP 端口；
- 游戏可以通过 IP:端口加入；
- 游戏服务端只监听本机、实体网卡或某个不方便暴露的地址；
- 需要像 TCP 代理一样把虚拟 IP 收到的数据转发到本地游戏端口。

### 2.2 基本原理

UDP 是无连接协议，没有 TCP 那样稳定的连接对象。代理需要维护“来源地址映射”：

```text
client_addr -> target_socket
```

流程：

1. 代理监听 `listen_addr:listen_port`。
2. 收到来自某个朋友 `client_addr` 的 UDP 数据。
3. 把数据转发到 `target_addr:target_port`。
4. 从目标游戏端口收到响应后，根据映射转发回对应 `client_addr`。
5. 映射需要设置超时时间，长时间无数据则清理。

### 2.3 MVP 功能

- 启动 UDP 代理：`listen_addr:listen_port -> target_addr:target_port`。
- 停止 UDP 代理。
- 查看运行状态：PID/任务 ID、监听地址、目标地址、包数、字节数、最近错误。
- 一键自测：启动临时 UDP echo 服务，启动临时 UDP 代理，发送 `hello udp proxy`，确认收到回包。
- 诊断报告加入 `udp_port_proxy_self_test` 检查项。

### 2.4 限制

- UDP 不保证到达顺序和可靠性；代理只能转发，不能保证游戏协议一定兼容。
- 某些游戏会把玩家公网/局域网地址写入协议负载，简单代理无法改写协议内容。
- 多玩家同时连接时必须正确维护来源地址映射。

## 3. UDP 广播桥

### 3.1 解决什么问题

UDP 广播桥主要解决“局域网房间发现”问题。

很多游戏寻找房间时不是直接连 IP，而是发送：

```text
255.255.255.255:端口
或
局域网广播地址:端口
或
特定组播地址:端口
```

在 n2n、Radmin、Tailscale 等虚拟网络中，广播/组播不一定按游戏预期穿透，导致：

```text
组网已经成功
IP 也能 ping 或端口能连
但游戏列表里看不到房间
```

UDP 广播桥的目标是把一侧收到的发现广播转发到另一侧或虚拟网络中，让游戏能“看见”房间。

### 3.2 与 UDP 端口代理的区别

| 能力 | 重点 | 典型用途 |
|---|---|---|
| UDP 端口代理 | 单播转发，朋友已经知道房主 IP/端口 | UDP 游戏端口直连 |
| UDP 广播桥 | 广播/组播发现包转发 | 游戏房间列表发现 |

UDP 广播桥不是简单的 `A端口 -> B端口`，它涉及：

- 本机网卡选择；
- 广播地址选择；
- 是否转发组播；
- 防止广播回环；
- 限制转发范围；
- 记录包来源，避免无限循环。

### 3.3 MVP 功能边界

第一版广播桥只做“受控、可诊断”的最小能力：

- 选择监听 UDP 端口；
- 选择本地物理网卡 / 虚拟网卡；
- 配置转发目标：虚拟网段广播地址或指定 peer IP；
- 显示收到的广播包数量、转发数量、丢弃数量；
- 防回环：同一包签名短时间内只转发一次；
- 不做协议伪装、不解析游戏私有协议。

### 3.4 发布级限制提示

UI 必须明确：

- 广播桥只能帮助“发现房间”，不等于保证加入成功；
- 加入成功仍取决于游戏实际端口、协议、组网、NAT、反作弊和账号系统；
- 对已支持 IP 直连的游戏，优先推荐直接连接虚拟 IP，不优先推荐广播桥；
- 广播桥未实现前，声明 `udp_broadcast_needed` 的游戏必须显示“能力待实现”。

## 4. 推荐实现顺序

1. UDP 单播端口代理后端：复用 TCP 代理的任务管理结构。
2. UDP 代理一键自测：临时 UDP echo 服务证明真实转发。
3. 通用组网中心 UI：新增 UDP 端口代理卡片。
4. 诊断报告：加入 `udp_port_proxy_self_test`。
5. adapter 推荐页：当 `requires_udp_broadcast_bridge=false` 且游戏需要 UDP 端口时，提示可试 UDP 端口代理。
6. UDP 广播桥文档与实验命令。
7. UDP 广播桥后端 MVP。
8. 广播桥诊断与适配器联动。

## 5. 验收标准

### UDP 端口代理

- 能通过一键自测真实收发 UDP 数据。
- 能显示包数、字节数、最近错误。
- 停止后释放端口。
- 诊断报告能说明 UDP 代理是否可用。

### UDP 广播桥

- 能捕获指定端口的广播包。
- 能向指定虚拟网卡或 peer 转发。
- 能避免明显回环风暴。
- 能在诊断报告中显示收到/转发/丢弃计数。
- 对用户明确说明“发现房间”和“成功加入”是两件事。
