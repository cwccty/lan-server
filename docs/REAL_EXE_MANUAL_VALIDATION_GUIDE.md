# 真实 EXE 人工验证指南

更新时间：2026-06-05

本指南用于 v0.1 发布前最后一轮人工验证。它不是开发环境测试，也不是只看 UI 状态变绿；每个 PASS 都必须来自真实 EXE、真实进程、真实端口、真实 n2n 日志、真实游戏内操作或诊断报告。

## 0. 验证前准备

在项目根目录先执行：

```powershell
npm run build
cargo check --manifest-path src-tauri\Cargo.toml
npm run adapter:validate
npm run release:preflight
```

如果需要重新生成发布产物：

```powershell
npm run tauri:build
powershell -ExecutionPolicy Bypass -File tools\real_exe_smoke_test.ps1 -StartupSeconds 5 -AppendLog
npm run release:package
```

真实 EXE 路径：

```text
src-tauri\target\release\lan-helper.exe
```

发布包路径：

```text
release-artifacts\v0.1.0\
```

## 2. A 组：真实 EXE 与基础体验

### A1. 启动干净

1. 双击或运行 `src-tauri\target\release\lan-helper.exe`。
2. 观察启动时是否出现白色命令框、透明残留窗口、多个主窗口。
3. 关闭程序，确认由联机助手启动的辅助进程不会异常残留。

通过标准：

- 主窗口标题为“联机助手”。
- 启动期没有额外 `cmd` / `conhost` / `powershell` 白框。
- `tools\real_exe_smoke_test.ps1` 输出 `status: PASS`。

### A2. 页面二次进入缓存

1. 依次打开：首页、游戏扫描、通用组网中心、Terraria 向导、推荐方案、适配器管理、诊断报告。
2. 第一次进入允许加载。
3. 第二次进入应优先展示上次内容，不应明显卡顿。
4. 如果真实后端刷新失败，应提示失败原因，而不是静默变灰。

通过标准：

- 游戏扫描保留上次扫描结果。
- 通用组网中心保留上次 n2n 参数。
- Terraria 向导保留世界、端口、密码、人数等表单。
- 诊断报告保留上次诊断结果。

## 3. B 组：好友邀请包一键加入

### B1. 房主生成邀请包

1. 在推荐方案页选择一个支持 LAN/IP 的游戏，例如 Terraria。
2. 启动 n2n 或确认 n2n 已在线。
3. 检测游戏端口。
4. 分配好友虚拟 IP。
5. 点击复制邀请包。

通过标准：

- 未分配好友 IP 时不能复制不完整 LAN 邀请。
- 邀请包包含房主虚拟 IP、端口、supernode、room/community、secret 或必要凭证。
- 邀请文案不能声称所有游戏都能一键联机。

### B2. 好友粘贴邀请包

1. 在另一台电脑，或单机模拟加入者场景，打开通用组网中心。
2. 粘贴房主邀请包。
3. 确认页面检测到邀请。
4. 点击“仅填入参数”。
5. 再点击“保存并启动 n2n”。

通过标准：

- “仅填入参数”只填表，不启动 edge。
- “保存并启动 n2n”会保存参数并启动 n2n。
- 页面出现加入结果卡片：成功、等待 ACK 或失败。
- 成功后提示：请在游戏内连接房主虚拟 IP 和端口。
- 失败时可以进入诊断，并能复制错误信息给房主。

### B3. 失败分类

至少模拟或观察以下失败之一：

- edge.exe 缺失；
- supernode 无响应；
- 房间密钥/room 不一致；
- 虚拟 IP 冲突；
- 游戏端口不通。

通过标准：

- 失败不是只显示 `false`。
- 诊断页能带入邀请上下文。
- 能复制包含失败原因、房主虚拟 IP、端口、supernode 和下一步建议的文本。

## 4. C 组：房主开房向导

### C1. Terraria 房主侧

1. 进入 Terraria 向导。
2. 选择世界或填写世界路径。
3. 设置端口，默认 `7777`。
4. 在程序内启动服务端。
5. 等待 30 秒。
6. 返回推荐方案页或诊断页检测端口。

通过标准：

- `7777` 或实际端口被监听。
- 服务端 30 秒内不自动退出。
- 诊断页能识别服务端状态。
- 推荐方案页能把“启动组网 → 启动服务端 → 检测端口 → 分配好友 IP → 生成邀请包”串起来。

### C2. 房主步骤条

在推荐方案页观察步骤条：

```text
选择游戏 → 推荐方案 → 启动组网 → 启动服务端/游戏 → 分配好友 IP → 生成邀请包
```

通过标准：

- 当前缺什么就提示做什么。
- n2n 状态和游戏端口状态来自真实检测。
- 不允许跳过关键条件生成误导性邀请包。

### C3. 游戏内加入

需要第二台电脑或虚拟机：

1. 房主保持 n2n 和 Terraria 服务端在线。
2. 加入者使用邀请包启动 n2n。
3. 加入者进入 Terraria。
4. 选择 Join via IP。
5. 输入房主虚拟 IP 和端口。

通过标准：

- 加入者能进入房主世界。
- 如果失败，诊断报告能说明是 n2n、端口、服务端、防火墙还是版本问题。

## 5. D 组：adapter 与非 LAN 转换边界

### D1. adapter registry 校验

执行：

```powershell
npm run adapter:validate
```

通过标准：

- `Adapters: 5` 或当前实际数量。
- `Warnings: 0` 或可解释。
- `Errors: 0`。
- `sha256` 与文件一致。

### D2. Cuphead / 本地同屏边界

1. 在方案库或游戏扫描中选择 Cuphead。
2. 查看推荐方案。

通过标准：

- 不生成 LAN/IP 邀请。
- 不要求 n2n 虚拟局域网。
- 推荐 Steam Remote Play 或 Sunshine + Moonlight。
- 明确说明本地同屏游戏不能真正变成 LAN。

## 6. E 组：诊断修复中心

1. 进入诊断报告页。
2. 分别在 n2n 未启动、supernode 无响应、端口不通、adapter 缺失或版本不匹配场景下生成诊断。
3. 点击可用的一键修复。
4. 执行自动复测。
5. 复制报告。

通过标准：

- 每个问题有原因说明。
- 有一键修复或手动修复命令。
- 修复后会自动复测。
- 报告能复制给朋友/管理员。
- 不把未知问题误写成已修复。

## 7. 记录方式

在真实 EXE 的诊断报告页，使用“真实 EXE 人工验证”区域逐项标记：

```text
PASS / FAIL / PENDING
```

然后点击复制验证清单，粘贴到：

```text
docs\RELEASE_VALIDATION_LOG.md
```

如果暂时只有一台电脑，可以把双机相关项保持为 `PENDING`，不要写成 PASS。

## 8. 发布判断

可以进入 v0.1 小范围测试的最低条件：

- A1/A2 PASS；
- B1/B2 至少单机模拟 PASS；
- C1/C2 PASS；
- D1/D2 PASS；
- E1 PASS；
- 双机游戏内加入如果未测，Release 文案必须保留 PENDING 边界。

不能发布或必须延后：

- 真实 EXE 启动有白框/透明残留窗口；
- n2n 状态经常误判；
- 未分配好友 IP 仍能复制 LAN 邀请；
- Cuphead 这类本地同屏被推荐为 n2n/LAN；
- 诊断失败只显示 `false` 或没有下一步。
