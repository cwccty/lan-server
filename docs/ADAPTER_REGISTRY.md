# 游戏适配器共享库方案

## 目标

用户不应该每次遇到一个游戏都手动填写复杂适配器。正确产品形态是：

> 少数用户、管理员或维护者完成一次游戏类型认定，生成适配器；后续其他用户扫描到同一个游戏时，客户端自动命中该适配器并直接给出转换方案。

也就是说，“新增游戏适配”不是普通玩家每次都要做的步骤，而是一个面向管理员、社区维护者和高级用户的贡献入口。

## 推荐架构

```text
客户端
├─ 内置适配器：随安装包发布，覆盖高频游戏
├─ 本地自定义适配器：高级用户/管理员手动创建
├─ 远程共享适配器库：官方或社区维护，客户端定期拉取
└─ 待审核适配器：用户提交后进入审核队列
```

## 用户体验

普通用户流程：

1. 扫描游戏。
2. 客户端用 Steam AppID / exe 名 / 路径特征匹配适配器。
3. 如果命中共享适配器，直接显示：
   - 是否可转换成本地联机
   - 推荐转换方式
   - 所需组件
   - 启动项 / 诊断项
4. 用户不需要理解适配器 JSON，也不需要手动判断游戏类型。

高级用户/管理员流程：

1. 选择一个未适配游戏。
2. 使用“适配器向导”认定游戏类型。
3. 本地生成 `custom_xxx.json`。
4. 可选择提交到共享库。
5. 管理员审核通过后，其他用户自动获得该适配器。

## 匹配优先级

当同一个游戏存在多个来源时，按以下顺序使用：

1. 本地用户显式覆盖适配器。
2. 官方审核通过适配器。
3. 社区可信适配器。
4. 安装包内置适配器。
5. 自动扫描出的未知 Steam 游戏。

## 适配器唯一标识

优先使用：

- Steam AppID
- 游戏 ID
- 主要 exe 文件名
- 默认端口
- 可选：安装目录特征文件

示例：

```json
{
  "game_id": "terraria",
  "steam_appid": "105600",
  "executables": ["Terraria.exe", "TerrariaServer.exe"]
}
```

## 共享库格式

远程共享库第一版可以是静态 JSON 索引，不必一开始做复杂后端：

```text
adapter-registry/
├─ index.json
└─ games/
   ├─ terraria.json
   ├─ minecraft_java.json
   └─ stardew_valley.json
```

`index.json` 包含版本、更新时间、适配器 hash：

```json
{
  "version": 1,
  "updated_at": "2026-06-02",
  "games": [
    {
      "game_id": "terraria",
      "steam_appid": "105600",
      "adapter_url": "games/terraria.json",
      "sha256": "..."
    }
  ]
}
```

## 安全和质量边界

共享适配器必须是数据，不是可执行代码：

- 不允许适配器携带任意脚本。
- 不允许自动下载未知 exe。
- 不允许绕过正版验证、反作弊或官方账号服务。
- 启动参数必须可展示、可确认、可回滚。
- 远程适配器需要 hash 校验，未来可加签名。

## 审核标准

管理员审核一个适配器时，至少确认：

1. 游戏身份是否准确。
2. 能力类型是否准确。
3. 转换方式是否真实可用。
4. 默认端口是否合理。
5. 是否存在误导性“一键联机”描述。
6. 是否涉及高风险操作。
7. 文案是否正常、无乱码。

## 开发顺序

1. 本地适配器管理页：查看、创建、编辑、保存。
2. 适配器导入/导出：让管理员能分享 JSON。
3. 远程 registry 拉取：客户端从固定 URL 更新共享适配器。
4. 适配器缓存与版本比较。
5. 用户提交入口：先导出文件，后续再做在线提交。
6. 管理员审核后台：后期再做，不进入 MVP。

## 产品结论

这个方案可以降低普通用户门槛：

- 普通用户只负责扫描和点击推荐流程。
- 高级用户/管理员负责少量游戏的类型认定。
- 适配器一旦进入共享库，后续所有用户都能复用。

因此，“适配器管理”应定位为管理员/高级功能，而不是普通玩家必经流程。
